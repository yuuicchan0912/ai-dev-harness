# 20 — TDD プロトコル

TDD は理想論ではなくループの状態遷移として強制する（`rules/10-loop-protocol.md`）。

## RED 証跡は必須

- 実装コードを書く前に、失敗するテストを書く。
- テストが実際に失敗することを確認し、そのログを
  `harness/evidence/<feature-id>/red.log` に保存する。
- RED 証跡がない実装は Evaluator が機械的に差し戻す。
- 実装後にテストが通ったログは `harness/evidence/<feature-id>/green.log` に保存する。

## テスト弱体化の禁止

- GREEN フェーズで、テストの expectation を緩める・削除する・`skip` / `todo` にして
  緑にすることを禁止する。
- カバレッジの数値目標ではなく、「契約のテスト観点表を全て満たすこと」を基準にする。

## テスト変更が許されるのは契約変更のときだけ

- 仕様が変わった場合は、その場でテストや実装を直さない。
- Planner に戻って feature-contract を更新し、契約変更としてテストを書き直す。
- 契約は凍結が原則。凍結後の変更は Planner 経由の明示的な契約更新に限る。

## フェーズ境界

- red: 失敗するテストのみを追加/変更する。
- green: プロダクトコードのみを変更してテストを通す。
- refactor: 挙動を変えずに整理する。テストコードの意味的変更はしない。

同一イテレーションでテストとプロダクトコードを両方書き換えて「緑になった」と報告しない。
