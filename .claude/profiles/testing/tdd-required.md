# testing profile — tdd-required

## 位置づけ

`rules/20-tdd-protocol.md` の運用詳細。TDD をループに組み込むための実務方針。

## 基本ループ

contract → red → green → refactor → evaluate（`rules/10-loop-protocol.md`）。
- red: 契約のテスト観点表からテストを書き、失敗ログを `red.log` に保存。
- green: 1 観点ずつ最小実装で通す。緑ログを `green.log` に保存。
- refactor: 挙動を変えずに整理。テストの意味は変えない。

## テストピラミッド方針

- 土台はユニット / コンポーネントテスト（速く多く）。
- 中間に統合 / 契約テスト（API 契約と実装の一致確認）。
- 頂点に E2E / スモーク（主要導線のみ、最小限）。

## カバレッジの基準

- 数値目標ではなく「契約のテスト観点表を全て満たすこと」を基準にする。
- 観点表に無いが重要なケースを見つけたら、契約（feature-contract）へ観点を追記してから
  テストを足す（Planner 経由）。

## 禁止

- RED 証跡なしの実装。
- テストの弱体化 / skip / todo による見かけの緑。
- テストとプロダクトコードを同一イテレーションで両方書き換えて緑にする行為。
