# 03 — TDD

TDD を理想論ではなくループの状態遷移として組み込みます（`rules/20-tdd-protocol.md` が正本）。

## 全体の流れ

```
仕様（feature-contract 凍結）
  → テスト（spec-to-tests で RED を確立）
  → 実装（1 観点ずつ GREEN）
  → 評価（Evaluator）
  → 修正（差し戻しに対応）
  → リファクタ（挙動不変で整理）
```

## RED を証跡化する

- 実装前に失敗するテストを書き、失敗を確認して `harness/evidence/<feature-id>/red.log` に保存。
- RED 証跡がない実装は Evaluator が機械的に差し戻す。
- GREEN 後は `green.log` を保存する。

## テスト弱体化の禁止

- expectation を緩める・削除する・`skip` / `todo` にして緑にすることを禁止。
- テスト変更が正当なのは契約変更（Planner 経由）のときだけ。

## フェーズ境界を守る

- red: 失敗テストのみ追加/変更。
- green: プロダクトコードのみ変更。
- refactor: 挙動を変えず整理。テストの意味は変えない。
- テストと実装を同一イテレーションで両方書き換えて「緑になった」と報告しない。

## カバレッジの考え方

- 数値目標ではなく「契約のテスト観点表を全て満たすこと」を基準にする。
- 重要な抜けを見つけたら契約に観点を追記してからテストを足す。

## スタック非依存にする仕掛け

- テスト / migration / lint / 依存監査のコマンドは skill に直書きしない。
- active-profile が指す profile の固定見出し（「テストコマンド」等）を参照する。
- これにより backend / frontend を切り替えても TDD 手順は変わらない。
